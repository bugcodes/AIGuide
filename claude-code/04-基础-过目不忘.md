### 2.1 为什么记忆系统如此重要？

想象一下，如果你每次和朋友聊天，他都不记得你们之前聊过什么——这将是一场灾难。Claude Code 同样如此。

**没有记忆系统的问题：**
```
用户：帮我重构登录模块
Claude：好的，请告诉我登录模块在哪里
用户：在 src/auth/login.py
Claude：好的，请告诉我登录模块的具体需求
用户：...（重复解释）
```

**有记忆系统后：**
```
用户：帮我重构登录模块（项目已有 CLAUDE.md）
Claude：好的，我了解你的项目使用 FastAPI + JWT，
        登录模块在 src/auth/login.py。
        我将保持现有的 JWT 认证逻辑不变，
        只重构代码结构。
```

### 2.2 Claude Code 的三层记忆架构

Claude Code 采用了类似人脑的记忆分层设计：

```
┌─────────────────────────────────────────────────────────┐
│                    第一层：会话记忆                       │
│         （当前对话，上下文窗口内，会话结束消失）             │
├─────────────────────────────────────────────────────────┤
│                    第二层：项目记忆                       │
│           （CLAUDE.md，绑定项目目录，永久保存）             │
├─────────────────────────────────────────────────────────┤
│                    第三层：持久记忆                       │
│      （用户偏好、跨项目知识，存储在用户配置目录）            │
└─────────────────────────────────────────────────────────┘
```

#### 第一层：会话记忆（Session Memory）

会话记忆是最短暂的，它的生命周期就是一个对话会话。

```python
# 会话记忆的特点：
session_memory = {
    "lifetime": "single conversation",
    "capacity": "约 200K tokens（上下文窗口）",
    "persistence": False,  # 会话结束即消失
    "content": [
        "当前对话的所有消息",
        "工具调用的历史结果",
        "当前任务的中间状态"
    ]
}
```

**会话记忆的工作方式：**

当你和 Claude Code 对话时，每一轮交互都会：
1. 将你的新输入追加到对话历史
2. 将 Claude 的回复追加到对话历史
3. 将工具调用结果追加到上下文
4. 如果上下文接近上限，Claude 会自动压缩早期内容

**会话记忆的实际表现：**

```
# 对话开始
> 你好，我是前端开发者，用 React + TypeScript

# Claude 会记住
会话记忆：你是一个前端开发者，使用 React + TypeScript

# 对话继续
> 帮我创建一个按钮组件

# Claude 理解你的技术栈后创建
按钮组件会使用 TypeScript 类型定义，符合 React 规范
```

#### 第二层：项目记忆（Project Memory）

项目记忆通过 `CLAUDE.md` 文件实现，是 Claude Code 记忆系统的核心。

```python
# 项目记忆的特点：
project_memory = {
    "lifetime": "permanent（直到文件被修改或删除）",
    "location": "项目根目录的 CLAUDE.md",
    "persistence": True,
    "scope": "仅当前项目",
    "loading": "自动加载，当 Claude 进入项目目录时"
}
```

**CLAUDE.md 的加载时机：**

```bash
# 当你在项目目录启动 Claude Code 时
cd /path/to/your/project
claude
# -> Claude 自动加载 ./CLAUDE.md

# 当你在项目目录内使用 --project 参数时
claude --project .
# -> Claude 自动加载 ./CLAUDE.md
```

#### 第三层：持久记忆（Persistent Memory）

持久记忆是跨会话、跨项目的长期记忆。

```python
# 持久记忆的位置：
persistent_memory_locations = {
    "global": "~/.claude/memory.md",        # 全局记忆
    "user_preferences": "~/.claude/config", # 用户偏好
    "skills": "~/.claude/skills/",          # 技能库
}
```

**全局记忆的使用场景：**

```markdown
<!-- ~/.claude/memory.md -->
# 我的编程偏好

## 语言优先级
1. Python（首选，数据处理）
2. TypeScript（前端开发）
3. Go（后端微服务）

## 编码规范
- 使用 4 空格缩进
- 行长度限制 120 字符
- 总是添加类型注解

## 常用工具
- Git（版本控制）
- Docker（容器化）
- PostgreSQL（数据库）
```

### 2.3 CLAUDE.md 核心配置详解

CLAUDE.md 是 Claude Code 项目记忆的核心。一个好的 CLAUDE.md 能让 Claude Code 真正理解你的项目。

#### 完整配置模板

```markdown

> [一句话描述项目是什么]

## 项目背景
[项目的背景故事，为什么要做这个项目]

## 技术栈
[列出项目使用的所有技术]

### 核心框架
- 框架1 + 版本
- 框架2 + 版本

### 编程语言
- Language + 版本

### 数据库
- 数据库类型 + 版本
- ORM/驱动版本

### 其他依赖
- 重要库和工具

## 项目结构
[详细的目录结构说明]

```
project/
├── src/
│   ├── components/    # UI 组件

└── docs/              # 文档

## 代码规范
[编码风格指南]

### 命名规范
- 组件：PascalCase（如 UserProfile.tsx）
- 工具函数：camelCase（如 getUserData.ts）
- 常量：UPPER_SNAKE_CASE

### TypeScript 规范
- 总是使用显式类型
- 禁止使用 any
- 优先使用 interface 而不是 type

### React 规范
- 使用函数组件 + Hooks
- 组件文件与样式文件放在一起

```bash

### 2.5 CLAUDE.md 的常见陷阱

#### 陷阱1：信息过长

**问题：** CLAUDE.md 超过 1000 行会让上下文污染严重。

**解决：** 保持简洁，只记录关键信息。使用链接指向详细文档。

```markdown
# Good
## 技术栈
- Next.js 14（参见 docs/tech-stack.md 了解详情）
- PostgreSQL 15
- Redis 7

# Bad
## 技术栈
- Next.js 14
  - React 框架
  - 支持 SSR 和 SSG
  - 最新的 App Router
  - ...
```

#### 陷阱2：信息过期

**问题：** 项目结构变化但 CLAUDE.md 没同步更新。

**解决：** 将 CLAUDE.md 更新纳入代码审查流程。

```bash
# 在 git hooks 中添加
# .git/hooks/pre-commit
git diff --cached CLAUDE.md && echo "CLAUDE.md 已修改"
```

#### 陷阱3：与 .claudeignore 混淆

**问题：** 不知道该用 CLAUDE.md 还是 .claudeignore。

**解决：** 记住这个原则：
- `CLAUDE.md` = 告诉 Claude 要记住什么
- `.claudeignore` = 告诉 Claude 要忽略什么

```markdown
# .claudeignore
# 不加载到上下文的文件
node_modules/
dist/
.env
*.log
```

#### 陷阱4：敏感信息泄露

**问题：** 在 CLAUDE.md 中写入了 API 密钥、密码等。

**解决：** 绝对不能在 CLAUDE.md 中写入任何敏感信息。

```markdown
# Good
## API 配置
API 密钥存储在 .env 文件中

# Bad
## API 配置
API_KEY=sk-xxxxyyyyzzzz
```

### 2.6 进阶技巧：CLAUDE.md 的高级用法

#### 技巧1：条件记忆

```markdown
## 不同场景的配置

# 当 task=refactor 时
重构时优先保持现有接口不变

# 当 task=new-feature 时
新功能需要先写文档和测试

# 当 task=bugfix 时
修复 bug 前先写测试用例
```

#### 技巧2：项目阶段标记

```markdown
## 项目阶段：[MVP] → [成长期] → [成熟期]

### MVP 阶段（当前）
- 核心功能已完成
- 文档不完善，可以跳过

### 成长期（计划中）
- 添加完整的单元测试
- 完善 API 文档
```

#### 技巧3：LLM 特定的提示

```markdown
## LLM 使用建议

### Claude Code 优先
- 使用 Claude Code 处理大型重构
- 处理多文件关联的修改

### GitHub Copilot 优先
- 简单的代码补全
- 单个文件内的小改动

### 组合使用
复杂任务先用 Claude Code 规划，再用 Copilot 实现
```

### 2.11 总结

> **核心要点回顾：**
> 1. Claude Code 有三层记忆系统：会话记忆、项目记忆、持久记忆
> 2. CLAUDE.md 是项目记忆的核心，让 Claude Code 理解你的项目
> 3. 一个好的 CLAUDE.md 应该包含：技术栈、项目结构、代码规范、常用命令
> 4. 避免 CLAUDE.md 过长、信息过期、包含敏感信息
> 5. 可以为不同项目类型创建标准化的 CLAUDE.md 模板
