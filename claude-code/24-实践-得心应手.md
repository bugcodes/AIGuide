### 22.2 分步骤执行复杂任务

很多人在让 Claude Code 做一个大项目时喜欢一步到位——"帮我写一个完整的电商系统"。这种做法往往导致：

```
问题1：上下文超载
Claude Code 的上下文窗口虽然很大，但也不是无限的。
一个完整电商系统的代码可能远超上下文容量。

问题2：输出质量下降
在单一回复中生成大量代码，会导致：
- 代码风格不统一
- 模块间耦合度高
- 缺乏统一的错误处理

问题3：难以调试和优化
一次性生成大量代码后，如果需要修改，
很难准确定位问题所在。
```

#### 22.2.1 正确的分解方法

**反面例子：**
```markdown
❌ "帮我写一个完整的电商系统，包括：
- 用户系统（注册、登录、OAuth）
- 商品系统（CRUD、分类、搜索）
- 订单系统（下单、支付、退款）
- 库存系统（库存扣减、预警）
- 物流系统（发货、物流跟踪）
- 促销系统（优惠券、满减、团购）
- 评价系统（打分、晒图、追评）
- 客服系统（工单、在线客服）
- 前后台管理系统"

这个任务太大了，一次性完成的质量会非常差。
```

**正面例子：**

```markdown
✅ 分 5 个阶段完成：

Phase 1: 架构设计（第1天）
"请为电商系统设计整体架构，包括：

"基于架构文档，实现：
- 项目脚手架（目录结构、配置文件）

"基于架构文档，实现用户模块：
- 用户注册/登录 API

### 22.3 真实案例深度剖析

#### 案例一：代码审查效率提升 300%

**背景：** 一个 15 人开发团队，每次 PR 审查平均耗时 45 分钟

**问题分析：**
```
- PR 数量多：每天 20-30 个 PR
- 审查时间长：平均每个 PR 需要 45 分钟
- 问题发现率低：经常遗漏安全问题
- 审查质量不稳定：不同人审查标准不一致
```

**Claude Code 介入方案：**

```python
# 建立代码审查 Multi-Agent 系统

"""
┌─────────────────────────────────────────────┐
│         PR 提交触发自动审查流程              │
├─────────────────────────────────────────────┤
│                                             │
│  1. GitHub Webhook 触发                    │
│     └─> 推送 PR 信息到 Claude Code         │
│                                             │
│  2. Orchestrator 分析 PR 类型               │

### 22.9 总结

> **核心要点回顾：**
>
> 1. **任务描述是核心技能**：十大模板覆盖 80% 日常场景
>
> 2. **分步骤执行是关键原则**：复杂任务必须分解，每个步骤验证后再继续
>
> 3. **真实案例是最好的老师**：从代码审查、文档自动化、Bug 定位到代码重构
>
> 4. **避免六类错误用法**：模糊描述、缺乏上下文、过度干预、不验证输出、上下文污染、忽略 Token 限制
>
> 5. **建立标准化工作流**：代码生成、Bug 修复、代码审查各有 SOP
>
> 6. **持续优化**：建立 Snippets 库、Preset 库，不断迭代
