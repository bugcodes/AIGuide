# Lesson 27 - 沙箱：镜像、自定义与工具策略（Images, Customization & Tool Policy）

## 核心概念

沙箱的镜像构建和工具策略决定了执行环境的完整性和安全性。本节涵盖镜像构建、沙箱浏览器、多 agent 覆盖和工具策略的交互。

---

## 关键设计

### 镜像构建

**默认镜像**：`openclaw-sandbox:bookworm-slim`（需要手动构建）
- 不包含 Node.js，需要通过 `setupCommand` 或自定义镜像安装

**通用工具镜像**：`openclaw-sandbox-common:bookworm-slim`
- 包含 curl、jq、nodejs、python3、git 等常用工具

**沙箱浏览器镜像**：`openclaw-sandbox-browser`
- 提供 Chromium 浏览器和 noVNC 访问（密码保护 + 短效 token URL）

### 沙箱浏览器关键配置

```json5
{
  agents: {
    defaults: {
      sandbox: {
        browser: {
          autoStart: true,
          autoStartTimeoutMs: 10000,
          network: "openclaw-sandbox-browser",  // 专用 Docker 网络
          cdpSourceRange: "172.21.0.1/32",       // CDP 入口允许列表
          allowHostControl: false,
          binds: ["/path/to/shared:/shared:ro"], // 独立于 docker.binds
        },
      },
    },
  },
}
```

Chromium 容器默认启动参数：
- `--disable-3d-apis`、`--disable-gpu`（无 GPU 时）
- `--no-sandbox`（当 `noSandbox` 启用时）
- `--renderer-process-limit=2`

### bind 挂载安全验证机制

OpenClaw 的路径验证不是简单的字符串匹配：
1. 规范化源路径
2. 通过最深层已存在的祖先重新解析
3. 重新检查阻塞路径和允许根目录

**这意味着 symlink 逃脱仍然会失败关闭**，即使最终路径不存在。

### 工具策略与沙箱交互

```
工具 deny/allow → 沙箱规则 → 结果
```

- 工具策略在沙箱规则之前评估
- 如果工具被全局或 per-agent deny，沙箱化不会让它复活
- `tools.elevated` 是明确的 escape hatch（绕过沙箱，使用 `gateway` 或 `node` escape path）

### 调试工具

```bash
openclaw sandbox explain
# 显示有效的沙箱模式、工具策略和修复配置键
```

### 多 Agent 覆盖

每个 agent 可以独立覆盖：
- `agents.list[].sandbox` - 沙箱配置
- `agents.list[].tools` - 工具 allow/deny
- `agents.list[].tools.sandbox.tools` - 沙箱工具策略

详见 [Multi-Agent Sandbox & Tools](/tools/multi-agent-sandbox-tools)。

### Docker-out-of-Docker (DooD) 约束

当 Gateway 本身在 Docker 容器中运行时：
- `openclaw.json` 的 `workspace` 配置必须使用**宿主机绝对路径**
- 路径必须完全相同（宿主机命名空间和容器内命名空间一致）
- 否则 OpenClaw 写入 heartbeat 时会报 `EACCES` 权限错误

## 自我检测

1. **问**：默认的 `openclaw-sandbox:bookworm-slim` 镜像包含 Node.js 吗？
   **答**：不包含。需要通过 `setupCommand` 安装或使用自定义镜像。

2. **问**：沙箱浏览器的 noVNC 访问默认如何保护？
   **答**：密码保护 + 短效 token URL（密码在 URL fragment 中，不在 query/header 日志中）。

3. **问**：如果有人在 symlink 逃脱后指向 `/etc/passwd`，OpenClaw 会拦截吗？
   **答**：会。路径验证会重新解析 symlink，即使最终路径不存在也会通过最深层已存在祖先检查并阻止。

4. **问**：工具策略和沙箱规则的执行顺序是什么？
   **答**：工具 deny/allow 策略先于沙箱规则评估。如果工具被 deny，沙箱化不会让它复活。

5. **问**：在 Docker 部署的 Gateway 中，workspace 配置有什么特殊要求？
   **答**：必须使用宿主机绝对路径，且需要在 Gateway 容器中以相同路径映射，使 Gateway 和 Docker daemon 对路径的解析一致。
