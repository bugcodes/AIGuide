# Lesson 35 - 安全实战：沙箱隔离与 bind 挂载防护

## 核心概念

沙箱安全的核心是理解隔离边界和 bind 挂载的风险。本节通过具体场景讲解如何正确配置沙箱以避免安全风险。

---

## 关键设计

### 隔离边界理解

**沙箱提供的是"减轻半径"而非"完美隔离"**。但它能有效防止模型做"蠢事"（如误操作文件系统）。

**不受沙箱保护**：
- Gateway 进程本身
- `tools.elevated` 明确的 escape hatch

**受沙箱保护**：
- 所有常规工具执行（exec、read、write、edit 等）

### bind 挂载风险矩阵

| 挂载类型 | 风险等级 | 说明 |
|----------|----------|------|
| `docker.sock` | 极高 | 允许容器逃逸 |
| `/etc`, `/proc`, `/sys`, `/dev` | 极高 | 系统级访问 |
| `~/.aws`, `~/.gnupg`, `~/.ssh` | 高 | 凭证泄露 |
| `/home/user/source:/source:ro` | 中 | 只读共享（需信任） |
| `workspace:rw` | 中 | 允许沙箱修改工作空间 |

### 路径验证深层机制

OpenClaw 的 bind 验证不是简单字符串匹配：
1. 规范化源路径（解析 symlink）
2. 通过最深层已存在祖先重新解析
3. 重新检查阻塞路径和允许根目录

**关键结果**：symlink 逃脱**仍然失败关闭**。

示例：
```
/workspace/run-link/new-file → /var/run/...
```
即使 `run-link` 指向 `/var/run/...`，如果 `/var/run/...` 在阻塞列表中，则拒绝。

### 安全配置最佳实践

**原则 1**：敏感挂载用 `:ro`（只读）
```json5
{
  docker: {
    binds: [
      "/home/user/shared:/shared:ro",  // 只读足够
      "/home/user/data:/data:ro",
    ],
  },
}
```

**原则 2**：结合 `workspaceAccess: "ro"` 使用
```json5
{
  sandbox: {
    workspaceAccess: "ro",
    docker: {
      binds: ["/mnt/cache:/cache:rw"],  // 仅必要时可写
    },
  },
}
```

**原则 3**：避免绑定 home 目录
```json5
// ❌ 危险
{ binds: ["/home/user:/home/user:rw"] }

// ✅ 安全
{ binds: ["/home/user/documents:/documents:ro"] }
```

### 网络安全配置

**默认**：无网络（`network: "none"`）

**可用选项**：
- `"bridge"`：标准桥接网络
- `"host"`：**被阻止**
- `"container:<id>"`**被阻止**（命名空间 join 绕过风险）
- 允许例外：`dangerouslyAllowContainerNamespaceJoin: true`（break-glass）

### 沙箱浏览器安全

- noVNC 默认密码保护 + 短效 token URL
- 密码在 URL fragment 中（不在 query/header 日志）
- `cdpSourceRange` 限制 CDP 入口
- `allowHostControl` 默认 false

## 自我检测

1. **问**：沙箱能提供完美的安全隔离吗？
   **答**：不能。文档明确说明"这不是完美的安全边界，但它能有效减少模型做蠢事时的损害半径"。

2. **问**：symlink 逃脱能否绕过 OpenClaw 的路径检查？
   **答**：不能。路径验证会规范化后重新解析，即使最终路径不存在也会通过最深已存在祖先检查并阻止。

3. **问**：bind 挂载 `~/.ssh` 到沙箱会怎样？
   **答**：OpenClaw 会阻止。`~/.ssh` 在常见凭证目录阻塞列表中。

4. **问**：如何最小化 bind 挂载的风险？
   **答**：使用 `:ro`（只读）；结合 `workspaceAccess: "ro"`；避免绑定整个 home 目录。

5. **问**：沙箱浏览器为什么用短效 token URL？
   **答**：减少 token 被截获的风险，且密码在 URL fragment 中而非 query/header 日志。
