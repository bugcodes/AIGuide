# Lesson 24 - 重试策略（Retry Policy）

## 核心概念

**重试策略**是 OpenClaw 对外向 provider 调用（如 Telegram、Discord）的容错机制。设计目标是：
- **按 HTTP 请求级别重试**，而非按多步流程重试
- **保持顺序性**：只重试当前失败步骤，不重做已完成步骤
- **避免重复**：对非幂等操作不轻易重试

---

## 关键设计

### 默认参数
| 参数 | 默认值 |
|------|--------|
| 最大尝试次数 | 3 |
| 最大延迟上限 | 30000 ms |
| 抖动（Jitter） | 0.1（10%） |
| Telegram 最小延迟 | 400 ms |
| Discord 最小延迟 | 500 ms |

### 重试触发条件

**模型 Provider（Stainless SDK 如 Anthropic、OpenAI）**
- 可重试响应：`408`、`409`、`429`、`5xx`
- 若 `retry-after-ms` 或 `retry-after` 超过 60 秒，OpenClaw 注入 `x-should-retry: false`，让 SDK 立即暴露错误，触发 model failover

**Discord**
- HTTP 429（rate limit）、请求超时、5xx 响应、瞬态传输失败（DNS 失败、连接重置、socket 关闭等）
- 优先使用 Discord `retry_after`，否则用指数退避

**Telegram**
- Transient errors：429、超时、连接重置、临时不可用
- 优先使用 `retry_after`，否则指数退避
- **Markdown 解析错误不重试**，降级为纯文本

### 配置示例

```json5
{
  channels: {
    telegram: {
      retry: {
        attempts: 3,
        minDelayMs: 400,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
    discord: {
      retry: {
        attempts: 3,
        minDelayMs: 500,
        maxDelayMs: 30000,
        jitter: 0.1,
      },
    },
  },
}
```

### 重要约束
- 重试应用于每次独立请求（消息发送、媒体上传、reaction、poll、sticker）
- 复合流程中已完成步骤不会被重试
- `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS` 可覆盖 SDK 长等待上限

## 自我检测

1. **问**：OpenClaw 重试是按"多步骤流程"还是"单次 HTTP 请求"为单位？
   **答**：按 HTTP 请求级别重试，保持顺序性。

2. **问**：Telegram 的 Markdown 解析错误会重试吗？
   **答**：不会，Markdown 解析错误不重试，会降级为纯文本发送。

3. **问**：当模型的 `retry-after` 超过多少秒时，OpenClaw 会注入 `x-should-retry: false`？
   **答**：超过 60 秒时注入，让 SDK 立即暴露错误，触发 model failover。

4. **问**：Discord 和 Telegram 重试时优先使用什么作为等待时间？
   **答**：优先使用平台自身的 `retry_after` 值（如 Discord 的 `retry_after` 字段），无法获取时才用指数退避。

5. **问**：如何让 SDK 完全自己处理重试（包括长等待）？
   **答**：设置 `OPENCLAW_SDK_RETRY_MAX_WAIT_SECONDS=0`（或 `false`、`off`、`none`、`disabled`）。
