# Lesson 14 - 模型故障转移 (Model Failover)

## 核心概念

OpenClaw 处理失败分两个阶段：
1. **Auth Profile 轮换**：在当前 provider 内轮换认证配置
2. **模型回退**：到 `agents.defaults.model.fallbacks` 中的下一个模型

## 关键设计

### 1. 候选链构建规则

- 请求的模型始终第一
- 显式配置的回退去重但不按模型 allowlist 过滤
- 如果当前运行已在同一 provider 族的配置回退上，继续使用完整配置链
- 如果当前运行在不同 provider 上，不追加其他 provider 的无关配置回退
- 未提供 `fallbacksOverride` 时，runner 在末尾附加配置的主模型
- 提供 `fallbacksOverride` 时，精确使用请求模型 + override 列表

### 2. Auth Profile 轮换顺序

1. 显式配置：`auth.order[provider]`
2. 配置的 profiles：`auth.profiles`
3. 存储的 profiles：`auth-profiles.json`

无显式顺序时，round-robin：
- **Primary key**：profile 类型（OAuth 优先于 API keys）
- **Secondary key**：`usageStats.lastUsed`（最旧优先）

### 3. Cooldown 机制

**Rate-limit/timeout bucket**：
- HTTP 429、Too many concurrent requests、ThrottlingException 等
- 也包括特定提供商的超时模式（如 `internal server error`、`unknown error, 520`）
- 使用指数退避：1分钟 → 5分钟 → 25分钟 → 1小时（上限）

**Billing disables**：
- "insufficient credits"/"credit balance too low"
- 标记为 **disabled**（更长退避）而非 short cooldown
- Billing backoff 从 **5小时** 开始，每失败翻倍，上限 **24小时**

### 4. Session Stickiness

- 选中的 auth profile **固定到会话**（保持 provider 缓存温暖）
- 直到：session reset、compaction 完成、profile 进入 cooldown
- 手动 `/model …@<profileId>` 设置用户覆盖，不会自动轮换直到新 session 开始

### 5. 回退持久化

在重试开始前，runner 将 selected fallback override 字段持久化到 session：
- `providerOverride`
- `modelOverride`
- `modelOverrideSource: "auto"`
- `authProfileOverride`
- etc.

这防止 failed fallback retry 覆盖更新的手动 session 变更。

### 6. 哪些错误会触发回退

**继续回退**：
- Auth failures
- Rate limits 和 cooldown exhaustion
- Overloaded/provider-busy 错误
- Timeout-shaped failover 错误
- Billing disables
- `LiveSessionModelSwitchError`

**不继续**：
- Context overflow 错误（应在 compaction/retry 内处理）
- 最终 unknown error 且无候选者

## 与其他课时的关联

- **Lesson 12**：每个 Agent 有自己的 model fallback 配置
- **Session 概念**：fallback 持久化解决 live reconciliation 竞态
- 与 `sessions_history` 的关系：recall 过滤敏感信息（credential-like text、tool call XML 等）
