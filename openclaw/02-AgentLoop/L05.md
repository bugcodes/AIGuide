# Lesson 05: 流式传输与分块

> OpenClaw 学习笔记 | 模块二：Agent Loop

## 核心概念

### 两种流式层

OpenClaw 有**两个独立的流式层**：

1. **Block Streaming（渠道消息）**：按完成的文本块发送，是普通渠道消息
2. **Preview Streaming（Telegram/Discord/Slack）**：更新临时预览消息

> 注意：目前没有真正的 token-delta 流式传输到渠道消息。Preview Streaming 是基于消息的（发送 + 编辑/追加）。

## 关键设计

### Block Streaming（块流）

```
Model output
  └─ text_delta/events
       ├─ (blockStreamingBreak=text_end)
       │    └─ chunker 按缓冲区增长发送块
       └─ (blockStreamingBreak=message_end)
            └─ 在 message_end 时刷新缓冲区
                   └─ 渠道发送（块回复）
```

**控制参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| `blockStreamingDefault` | `"on"`/`"off"` | 默认 off |
| `blockStreamingBreak` | `"text_end"` / `"message_end"` | 刷新时机 |
| `blockStreamingChunk` | `{minChars, maxChars, breakPreference}` | 块大小 |
| `blockStreamingCoalesce` | `{minChars?, maxChars?, idleMs?}` | 合并块 |
| `humanDelay` | `off`/`natural`/`custom` | 块间暂停 |

### 分块算法

`EmbeddedBlockChunker` 实现：

- **Low bound**：不到 `minChars` 不发送（除非强制）
- **High bound**：优先在 `maxChars` 前分割；强制时在 `maxChars` 处分割
- **Break preference**：`paragraph` → `newline` → `sentence` → `whitespace` → hard break
- **Code fences**：绝不在 fence 内分割

### 合并（Coalescing）

合并连续块后再发送，减少"单行刷屏"：

- 等待空闲间隙（`idleMs`）后刷新
- 缓冲区被 `maxChars` 上限限制
- `minChars` 防止发送过小片段

### Preview Streaming 模式

| 模式 | 说明 |
|------|------|
| `off` | 禁用预览流 |
| `partial` | 单个预览，被最新文本替换 |
| `block` | 预览分块/追加更新 |
| `progress` | 生成期间的状态预览，完成时最终答案 |

### 渠道映射

| 渠道 | off | partial | block | progress |
|------|-----|---------|-------|----------|
| Telegram | ✅ | ✅ | ✅ | 可编辑进度草稿 |
| Discord | ✅ | ✅ | ✅ | 可编辑进度草稿 |
| Slack | ✅ | ✅ | ✅ | ✅ |
| Mattermost | ✅ | ✅ | ✅ | ✅ |
| MS Teams | ✅ | ✅ | ✅ | 本机进度流 |

### 工具进度预览

Preview Streaming 可以包含**工具进度**更新（如"搜索网络"、"读取文件"），出现在工具运行期间的预览消息中。

支持渠道（默认启用）：Discord、Slack、Telegram、Matrix

## 输出

通过本次学习，我掌握了 OpenClaw 的流式传输设计：

1. **双层分离**：Block Streaming 用于消息，Preview Streaming 用于预览
2. **分块控制**：灵活的字符限制和分割偏好
3. **合并优化**：减少刷屏，保持实时性
4. **渠道差异**：不同渠道有不同的流式支持
5. **工具进度**：实时显示工具运行状态

流式传输让 OpenClaw 的响应更实时，是良好用户体验的关键。
