# Lesson 29 - Plugin Manifest 与插件清单配置

## 核心概念

Plugin Manifest（`openclaw.plugin.json`）是每个 plugin 必须提供的声明文件，用于声明 plugin 的 ID、能力合约、激活方式、配置 schema 等。OpenClaw 据此发现和加载 plugin runtime。

---

## 关键设计

### Manifest 完整结构

```json
{
  "id": "my-plugin",
  "name": "My Plugin",
  "description": "描述 plugin 功能",
  "version": "1.0.0",
  "contracts": {
    // 发现 + 所有权声明
    "tools": ["my_tool", "workflow_tool"],
    "agentToolResultMiddleware": ["my-middleware"]
  },
  "activation": {
    "onStartup": true
  },
  "toolMetadata": {
    "workflow_tool": {
      "optional": true
    }
  },
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {
      "apiKey": { "type": "string" }
    }
  }
}
```

### 核心字段

**id**：唯一标识符，用于发现和配置

**contracts**：声明 plugin 提供的所有能力
- `tools`：注册的工具列表（必须与 runtime 注册一致）
- `agentToolResultMiddleware`：tool 结果中间件（仅限受信任的 bundled plugin）

**activation.onStartup**：是否在启动时激活，plugin 应明确设置

**toolMetadata**：工具的额外元数据
- `optional: true` - 用户必须主动加入 allowlist 才加载
- 用户通过 `tools.allow: ["workflow_tool"]` 或 `tools.allow: ["my-plugin"]` 启用

**configSchema**：JSON Schema 格式的配置验证

### 关键原则

1. **Runtime 注册的工具必须在 contracts.tools 中声明**，这样 OpenClaw 可以发现所有权而不必加载每个 plugin runtime
2. OpenClaw 捕获并缓存已注册工具的验证描述符，manifest 中不需要重复 `description` 或 schema
3. manifest contract 仅声明所有权和发现；执行时调用 live registered tool 实现
4. `toolMetadata.<tool>.optional: true` 对应 `api.registerTool(..., { optional: true })`

### package.json 中的 openclaw 元数据

```json
{
  "name": "@myorg/openclaw-my-plugin",
  "version": "1.0.0",
  "type": "module",
  "openclaw": {
    "extensions": ["./index.ts"],
    "compat": {
      "pluginApi": ">=2026.3.24-beta.2",
      "minGatewayVersion": "2026.3.24-beta.2"
    },
    "build": {
      "openclawVersion": "2026.3.24-beta.2",
      "pluginSdkVersion": "2026.3.24-beta.2"
    }
  }
}
```

### ClawHub 发布检查清单

1. package.json 有正确的 `openclaw` 元数据
2. `openclaw.plugin.json` manifest 存在且有效
3. 入口使用 `defineChannelPluginEntry` 或 `definePluginEntry`
4. 所有导入使用聚焦的 `plugin-sdk/<subpath>` 路径
5. 内部导入使用本地模块，不是 SDK self-imports
6. 测试通过

## 自我检测

1. **问**：为什么 `contracts.tools` 是必须的？
   **答**：让 OpenClaw 发现 plugin 所有权而不必加载每个 plugin runtime。运行时注册的 tool 必须列出以便发现。

2. **问**：manifest 中的 tool schema 与 runtime 注册的 tool 是什么关系？
   **答**：Manifest contract 仅声明所有权和发现。OpenClaw 捕获并缓存 runtime 注册的 validated descriptor，不重复 manifest 中的 description 或 schema。

3. **问**：可选工具（optional tool）需要在哪里声明？
   **答**：在 `toolMetadata.<tool>.optional: true` 中声明，对应 runtime 的 `api.registerTool(..., { optional: true })`。

4. **问**：用户如何启用 plugin 的可选工具？
   **答**：通过 `tools.allow: ["workflow_tool"]` 启用单个工具，或 `tools.allow: ["my-plugin"]` 启用 plugin 的所有工具。

5. **问**：内部导入不应该使用什么路径？
   **答**：不应该使用 SDK self-imports（如 `from "openclaw/plugin-sdk"`），应使用本地模块路径。
