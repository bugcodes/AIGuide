# Lesson 04: 消息流程与会话管理

> OpenClaw 学习笔记 | 模块二：Agent Loop

## 核心概念

### 消息流程概览

```
入站消息
  -> 路由/绑定 -> session key
  -> 队列（如果正在运行）
  -> agent run（流式 + 工具）
  -> 出站回复（渠道限制 + 分块）
```

### 消息 vs 工具结果

- **`toolResult.content`**：模型可见的结果
- **`toolResult.details`**：运行时元数据（UI 渲染、诊断、媒体传递、插件用）

> 关键：模型必须读取的内容放在 `content`，不要只放在 `details`

## 关键设计

### 入站去重

渠道可能在重连后重新传送同一消息。OpenClaw 维护短期缓存，按 `channel/account/peer/session/message id` 索引，避免重复触发 agent run。

### 入站防抖

同一发送者的快速连续消息可以合并为单个 agent turn。

```json5
{
  messages: {
    inbound: {
      debounceMs: 2000,
      byChannel: {
        whatsapp: 5000,
        slack: 1500,
        discord: 1500,
      },
    },
  },
}
```

- 防抖仅适用于**纯文本**消息；媒体/附件立即刷新
- 控制命令绕过防抖（除非渠道明确选择 DM 合并）

### Body 分层

OpenClaw 区分消息的不同部分：

| 类型 | 用途 |
|------|------|
| `BodyForAgent` | 当前消息的主要模型输入 |
| `Body` | 遗留 prompt fallback |
| `CommandBody` | 用于指令/命令解析的原始用户文本 |
| `RawBody` | `CommandBody` 的遗留别名 |

### 历史缓冲

当渠道提供历史时，使用共享包装：

```
[Chat messages since your last reply - for context]
[Current message - respond to this]
```

- 非直接聊天（群组/频道），当前消息前加发送者标签
- 历史缓冲是**仅待处理**的：包含未触发 run 的群组消息，排除已存在于 session transcript 的消息
- 指令剥离仅适用于**当前消息**部分

### 会话与设备

- 会话由 Gateway 拥有，不是客户端
- 直接聊天合并到 agent main session key
- 群组/频道获得自己的 session key
- 多设备/渠道可以映射到同一会话，但历史不会完全同步到每个客户端

### 队列和 Followup

如果 run 已激活，入站消息可以：

- **steer**：转向当前运行
- **followup**：收集等待后续 turn
- **collect**：收集多个消息
- **interrupt**：中断当前运行

### 静默回复

`NO_REPLY` / `no_reply` 精确 token 意味着"不发送用户可见回复"。

但如果 turn 有待处理的工具媒体（如生成的 TTS 音频），OpenClaw 仍会传递媒体附件。

### 推理可见性

- `/reasoning on|off|stream` 控制可见性
- 推理内容仍计入 token 使用
- Telegram 支持推理流到临时草稿气泡，最终传递后删除

## 输出

通过本次学习，我理解了 OpenClaw 的消息处理管道：

1. **多阶段处理**：从入站到出站的完整流程
2. **Body 分层**：不同用途的消息部分
3. **去重+防抖**：防止重复和过度触发
4. **会话隔离**：Gateway 拥有会话，不是客户端
5. **智能队列**：多种模式处理并发消息

消息流程是 Agent Loop 的入口，理解它对于配置消息行为至关重要。
