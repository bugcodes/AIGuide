# Lesson 25 - 沙箱基础：模式与范围（Sandboxing: Modes & Scope）

## 核心概念

**沙箱（Sandboxing）**是 OpenClaw 的可选安全特性，用于在隔离的后端运行工具，减少模型误操作导致的"爆炸半径"。沙箱仅作用于**工具执行**（exec、read、write、edit 等），Gateway 进程本身始终运行在宿主机上。

---

## 关键设计

### 什么被沙箱化

✅ **被沙箱化**：
- 工具执行：`exec`、`read`、`write`、`edit`、`apply_patch`、`process` 等
- 可选的沙箱浏览器（`agents.defaults.sandbox.browser`）

❌ **不被沙箱化**：
- Gateway 进程本身
- 明确配置为 `tools.elevated` 的工具（绕过沙箱，使用配置的 escape path）
- 沙箱关闭时，所有工具都在宿主机上运行

### 三种模式（mode）

| 模式 | 含义 |
|------|------|
| `"off"` | 不使用沙箱（默认） |
| `"non-main"` | 仅沙箱化**非 main**会话（默认推荐） |
| `"all"` | 所有会话都在沙箱中运行 |

> `non-main` 基于 `session.mainKey`（默认 `"main"`）判断，而非 agent id。群组/频道会话使用独立 key，因此会被沙箱化。

### 三种范围（scope）

| 范围 | 含义 |
|------|------|
| `"agent"`（默认） | 每个 agent 一个容器 |
| `"session"` | 每个 session 一个容器 |
| `"shared"` | 所有沙箱会话共享一个容器 |

### 后端类型（backend）

| 后端 | 说明 |
|------|------|
| `"docker"`（默认） | 本地 Docker 容器，适合本地开发 |
| `"ssh"` | 通用 SSH 远程沙箱，可将执行卸载到 SSH 可访问的任意主机 |
| `"openshell"` | OpenShell 托管的远程沙箱，支持可选的双向同步 |

## 自我检测

1. **问**：沙箱化和关闭沙箱，Gateway 进程本身在哪里运行？
   **答**：Gateway 进程始终在宿主机上运行。沙箱只影响工具执行的环境。

2. **问**：`mode: "non-main"` 是基于什么判断一个会话是否要沙箱化？
   **答**：基于 `session.mainKey`（默认 `"main"`），而非 agent id。群组/频道会话使用独立 key，会被沙箱化。

3. **问**：`tools.elevated` 工具会经过沙箱吗？
   **答**：不会。`tools.elevated` 是明确的 escape hatch，绕过沙箱直接使用配置的 escape path（默认 `gateway`）。

4. **问**：Docker 后端沙箱默认使用哪个 Docker 镜像？
   **答**：`openclaw-sandbox:bookworm-slim`，需要通过 `scripts/sandbox-setup.sh` 或手动 `docker build` 构建。

5. **问**：SSH 后端和 OpenShell 后端都支持沙箱浏览器吗？
   **答**：都不支持。沙箱浏览器仅在 Docker 后端上受支持。
