# Lesson 36 - 安全实战：工具策略与沙箱交互

## 核心概念

工具策略（Tool Policy）和沙箱是两道独立的安全防线。理解它们的交互关系是安全配置的关键。

---

## 关键设计

### 执行顺序（重要！）

```
工具 deny/allow → 沙箱规则 → 执行结果
```

1. **工具策略先评估**：如果工具被全局或 per-agent deny，沙箱化不会让它复活
2. **沙箱规则后执行**：工具策略放行后，沙箱才介入决定执行位置（宿主机或容器）

### 工具策略配置

```json5
{
  tools: {
    deny: ["exec"],          // 禁用 exec
    allow: ["read", "write"], // 白名单模式
  },
}
```

per-agent 覆盖：
```json5
{
  agents: {
    list: [
      {
        id: "build",
        tools: {
          deny: ["exec"],
          sandbox: {
            tools: {
              deny: ["read"],  // 沙箱内额外禁用 read
            },
          },
        },
      },
    ],
  },
}
```

### elevated 工具的特殊性

`tools.elevated` 是明确的 escape hatch：
- 绕过沙箱
- 使用配置的 escape path（默认 `gateway`，或 `node` 当 exec target 是 `node`）
- **不受沙箱模式影响**
- **仍受工具策略 deny 影响**（工具策略优先）

### /exec 指令的特殊性

`/exec` 指令：
- 仅对授权发送者生效
- 每会话持久化
- 可通过工具策略 deny 完全禁用

### 沙箱工具策略

```json5
{
  agents: {
    list: [
      {
        id: "code-agent",
        sandbox: {
          tools: {
            deny: ["write", "edit"],  // 沙箱内禁用写入工具
          },
        },
      },
    ],
  },
}
```

### 多层安全覆盖优先级

```
全局 tools deny → per-agent tools deny → per-agent sandbox tools deny
```

更深层的配置可以加严但不能放松（更深的 deny 优先于更浅的 allow）。

### openclaw sandbox explain 输出解读

运行 `openclaw sandbox explain` 可看到：
- 有效沙箱模式
- 工具策略（全局 + per-agent）
- 沙箱内工具策略
- elevated 配置
- 修复建议

### 安全配置检查流程

```
1. 工具策略允许该工具吗？ → 否 → 直接拒绝
2. 工具是 elevated 吗？ → 是 → 使用 escape path
3. 沙箱模式是什么？ → off → 宿主机执行
4. 沙箱工具策略允许吗？ → 否 → 沙箱内拒绝
5. → 沙箱内执行
```

## 自我检测

1. **问**：如果全局工具策略 deny 了某工具，沙箱化能让它复活吗？
   **答**：不能。工具策略在沙箱规则之前评估，deny 是终局的。

2. **问**：`tools.elevated` 是否受沙箱模式影响？
   **答**：不受。`tools.elevated` 是明确 escape hatch，使用 escape path 而非沙箱。

3. **问**：`tools.elevated` 是否受工具策略影响？
   **答**：是。工具策略 deny 优先于沙箱，elevated 无法绕过 deny。

4. **问**：per-agent sandbox tools deny 和全局 tools deny 的优先级是什么？
   **答**：更深层的 deny 优先。更深的配置可以加严但不能放松。

5. **问**：如何完全禁用 exec（包括沙箱内的）？
   **答**：在全局工具策略中 `tools.deny: ["exec"]`，这会在沙箱规则之前拦截。
